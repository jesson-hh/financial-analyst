# 观澜 · 落子 —— 实盘交易引擎设计(研判循环 · 影子组合 · 校场策略)

- 日期:2026-06-08
- 模块:`ui/seats/`(前端)+ `guanlan_v2/seats/`(后端薄壳)+ `engine/financial_analyst/`(引擎)
- 状态:设计已与用户逐条确认,待评审 → 出实施计划

---

## 1. 背景与动机

当前「落子」的实盘(live)模式存在三处**名实不符 / 未定义**,均由用户在使用中发现:

1. **实盘合议 = 复盘合议(同一份历史回测)**。`consensus` 只依赖 `symbol`(历史 K 线 + 各席位历史决策)与启用席位,与 `mode` 无关([luozi-app.jsx:66](../../ui/seats/luozi-app.jsx))。两模式仅换标签(实盘合议/回测合议),数字逐位相同 →「实盘合议 +5.9%」其实是历史回测,不是实盘真绩效。
2. **交易策略无定义**:实盘下「切换时间尺度时 LLM 按哪个周期判断」「何时生成交易单」「多久出一次信号」「中长期还是短线」全无规定。
3. (已在本轮单独修掉的同类问题:市况「主线」硬编 mock、实盘日期显示历史末根日而非今日——见 `ui/seats/README.md` 对应条目。)

根因:**实盘从未被定义成一个"向前运行的真实交易系统"**——没有交易策略口径、没有实时决策节拍、没有实盘持仓跟踪,于是只能复用历史回测并贴上"实盘"标签。

本设计的目标:把实盘定义成一个**诚实、向前运行**的系统,使实盘绩效有真实含义、与复盘有清晰分界。

---

## 2. 核心设计决策(均经用户确认)

### 2.1 时间尺度(TF)= 纯看盘镜头
图表的 TF 选择器(周/日/60/30/15/5/1 分)**只改变你看到的粒度,不改变任何策略**。每个席位按它在校场装配出的**自有周期**判断与交易;切到 1min 不会让一个中期席位变成超短。LLM 研判用**席位周期**,不用图表 TF。

### 2.2 席位 = 用户自命名、按票配置的"策略实例"
- 固定 4 席(动量/反转/事件/风控)降级为**模板/起点**,不再是唯一选项。
- 在**校场(Foundry)** 把 `经验卡 + 因子 + 研报 + 周期参数` 组装成一个**新策略,用户自己起名**(如「宁德·突破回踩」)。
- **按票配置**:每只关注票拥有自己的一套命名策略;该票的"合议" = 它这套策略的合议。
- 数据模型(策略实例):
  ```
  StrategyInstance = {
    id,                       // 稳定主键(泛化用,绝不写死 4 席)
    name,                     // 用户自命名
    recipe: { cards[], factors[], research[] },   // 配方(共享库引用)
    clock: {                  // 周期参数 = 交易时钟(校场显式设定)
      execTF,                 // 决策所在 K 线:day | 5min(≠ 看盘 TF)
      decisionFreq,           // 决策频率(多久重判一次)
      maxHold,                // 最长持有(bar 数)→ 时间止盈
      stopLoss, takeProfit    // 止损 / 止盈 %
    },
    bind: [code, ...]         // 绑定的标的(按票配置)
  }
  ```

### 2.3 席位周期来源 = 校场显式设定 + 演武验证
装配席位时由用户**显式设定** `clock`(execTF / decisionFreq / maxHold / stopLoss / takeProfit)。校场"演武"(跨标的回测)**按这组周期参数**回测,验证该策略在此周期下成不成立(并回显实测平均持有期供校验)。周期是用户定的、数据验证它。

### 2.4 研判循环(LLM 思考层,三触发穿插)
"研判" = 一次 LLM 思考。由三类事件穿插触发:
1. **手动**:用户随时点「研判」(现已具备)。
2. **事件:条件单成交 → 触发下一次研判**。某条件单触发成交(进场/出场)后持仓状态变化,自动启动下一轮研判:继续持 / 加仓 / 挂下一个止盈止损单。
3. **定时:盘中周期研判**。交易时段定时心跳。**节流规则:以「每小时」为封顶(最快每小时一次),席位周期更慢就按更慢**(日线席 → 每日收盘一次,不每小时空跑;盘中席 → 每小时)。保证 LLM 成本有上限又尊重席位周期。

### 2.5 两层架构
- **研判层(LLM · 贵 · 慢)**:由 2.4 三触发驱动,产出"意图"——挂/改/撤一张条件单 + 持仓建议。复用现有 `/seats/order`(条件单生成)+ `/seats/live_eval`(实时上下文)。
- **执行层(条件单引擎 · 便宜 · 秒级)**:持续盯价格/量比/指标,触发即**提示用户手机下单**。复用现有 `evalTrigger` / 触发引擎 + `OrderWatchPanel` 实时盯盘。

> 红线沿用:**系统只出信号,不代下单**;一切由用户在手机端手动执行。

### 2.6 B —— 实盘绩效 = B1 影子组合(诚实标注)
因系统不代下单、不知道用户真实成交,实盘绩效采用 **B1 影子组合**:
- **自上线日(用户点"开始跟踪"那天)起**,每次条件单触发 → 按**信号价**记入影子持仓台账,按 `(策略, 票)` 跟踪虚拟盈亏。
- 算实盘指标(累计/回撤/胜率/Sharpe)**只在影子组合上、自上线日起**。
- **必须诚实标注**:「信号绩效 · 影子组合 · 自 YYYY-MM-DD 起 · 未计真实成交/滑点」。**绝不**冒充用户真实账户绩效,**绝不**复用历史回测。
- B2(回填真实成交 / 接券商 API 升级为真实账户绩效)**本期不做**,留作后续叠加层。

**最终复盘 / 实盘分界(本设计的核心收获):**

| | 含义 | 数据源 | 起算点 |
|---|---|---|---|
| **复盘合议** | 策略在历史全程的**回测**绩效 | 历史 K 线 + 历史决策 | 数据起点 |
| **实盘合议** | 策略的**影子组合(信号)绩效** | 上线后实时触发的条件单(信号价) | 上线日 |

## 3. 架构与数据流

```
校场(装配命名策略 + 周期参数 + 绑票)──演武按周期回测验证──▶ 上线名册(按票)
        │
        ▼
盯盘/实盘(按票)
  研判触发器 {手动 | 条件单成交后 | 定时·每小时封顶·席位周期取慢}
        │ (LLM 研判层:读配方+实时上下文 → 意图)
        ▼
  生成/改/撤 条件单 ──▶ 触发引擎(秒级盯价/量比/指标)
        │                        │ 触发
        │                        ▼
        │                   提示用户手机下单(系统不代下单)
        ▼                        │
  影子组合台账 ◀────── 记一笔(信号价, 自上线日) ─────┘
        │ (按 (策略,票) 跟踪虚拟持仓/盈亏)
        ▼
  实盘合议指标(自上线日起算,诚实标注) + 复盘合议(历史回测)分列
```

### 关键不变量 / 红线
- **展示帧 vs 计算帧分离**(沿用现有红线):影子组合台账与合议计算只吃真实触发记录;图表显示帧(forming bar 等)绝不污染台账。
- **诚实降级**:无后端 / 无信号 / 无成交 → 显示 `—` 或明确空态,**绝不编造**(沿用本轮"主线/涨跌停/日期"去 mock 的纪律)。
- **泛化策略 id**:第 1、2 期一律按 `策略 id` 设计,不写死 4 席;第 3 期只替换"策略来源",不返工。

## 4. 实施分期(每期独立可用、可验证)

### 第 1 期 · 研判循环(让实盘"会动")
在现有 条件单/LLM 上接齐三触发器:手动(已有)+ **条件单成交后自动研判** + **盘中每小时封顶的定时研判**(席位周期取慢)。
- 触点:`OrderWatchPanel`(luozi-panels)live 监控扩成"成交→研判"事件;新增定时心跳调度;复用 `/seats/order` `/seats/live_eval`。
- 验收:实盘下,触发成交后自动产生下一轮研判;定时按每小时封顶节拍研判;LLM 调用量受控;0 console error;浏览器实测。

### 第 2 期 · B1 影子组合(让实盘绩效真且诚实)
建持仓台账,把每次条件单触发按信号价记成影子持仓(自上线日起),算实盘指标,**替换"实盘=历史回测"的假合议**,复盘/实盘分界 + 诚实标注。
- 触点:新增影子组合状态/存储(前端 state + 可选持久化);`MetricsStrip` 实盘分支改吃影子组合指标(复盘分支不变);标签/起算点/空态诚实化。
- 验收:实盘合议数字 ≠ 复盘;自上线日起算;每笔可溯源到触发记录;诚实标注齐全;浏览器实测。

### 第 3 期 · 校场策略升级(自命名 · 按票配置)
固定 4 席 → 用户自组装命名的策略实例 + 按票绑定 + 演武按周期回测。**最大改动,放最后。**
- 触点:`Foundry`(luozi-foundry)装配 UI 扩"命名 + 周期参数 + 绑票";`LZ_SEATS`/`perSeat`/`scanSeat`/`seatColor` 等从"固定 4 席"泛化为"策略实例集";演武按 `clock` 回测。
- 验收:可新建命名策略、设周期、绑票、演武出按周期的真回测;前两期逻辑无返工接入;浏览器实测。

> 顺序:先 **1 → 2**(价值最高:实盘既会动、绩效又诚实,直接解决用户提出的两个问题),**3** 随后单独立项(自身的 spec → plan → 实施)。
